# 기술스택 정리

## 프로젝트 기준 기술스택
- Frontend: React + Vite
- Backend: Node.js + Express
- Database: MongoDB Atlas + Mongoose
- API 통신: Axios
- 환경변수 관리: dotenv
- CORS 처리: cors
- 개발 편의: nodemon
- 스타일링: CSS 또는 Tailwind CSS
- 배포: Vercel
- 버전 관리: Git + GitHub

## 각 기술이 왜 필요한가

### 1. React
- 역할: 사용자에게 보여지는 화면(UI)을 만드는 프론트엔드 라이브러리
- 왜 필요한가:
- Todo 입력창, 버튼, 목록, 체크박스 같은 화면 요소를 만들기 위해 필요하다.
- 데이터가 바뀌었을 때 화면을 다시 보여주는 작업을 편하게 처리할 수 있다.
- 컴포넌트 단위로 나눠서 만들 수 있어서 유지보수가 쉽다.

### 2. Vite
- 역할: React 프로젝트를 빠르게 생성하고 개발 서버를 실행하는 도구
- 왜 필요한가:
- React만으로는 프로젝트 실행 환경이 바로 준비되지 않기 때문에 Vite가 필요하다.
- 개발 서버 실행이 빠르고 설정이 비교적 단순하다.
- 초보자가 React 프로젝트를 시작하기에 편하다.

### 3. Node.js
- 역할: JavaScript를 브라우저 밖, 즉 서버에서도 실행할 수 있게 해주는 런타임
- 왜 필요한가:
- Express 서버를 실행하려면 Node.js가 필요하다.
- 프론트와 백엔드를 같은 JavaScript 계열로 학습할 수 있다.

### 4. Express
- 역할: 백엔드 서버와 API를 만들기 위한 Node.js 프레임워크
- 왜 필요한가:
- Todo 데이터를 조회, 추가, 수정, 삭제하는 API를 만들기 위해 필요하다.
- 요청을 받고 응답을 보내는 서버 로직을 단순하게 작성할 수 있다.
- CRUD 구조를 배우기에 적합하다.

### 5. MongoDB Atlas
- 역할: 클라우드에서 MongoDB 데이터베이스를 제공하는 서비스
- 왜 필요한가:
- Todo 데이터를 저장해 두기 위해 필요하다.
- 로컬 파일이 아니라 실제 데이터베이스에 저장하므로 새로고침하거나 다시 접속해도 데이터가 유지된다.
- Vercel에 배포한 뒤에도 외부에서 접근 가능한 DB가 필요하므로 Atlas가 적합하다.

### 6. Mongoose
- 역할: Express와 MongoDB를 쉽게 연결하고 데이터를 다루게 해주는 라이브러리
- 왜 필요한가:
- MongoDB에 직접 쿼리를 보내는 것보다 더 쉽게 데이터 모델을 정의할 수 있다.
- Todo의 형태를 스키마로 정리할 수 있다.
- 예를 들어 `title`, `completed` 같은 필드를 명확하게 관리할 수 있다.

### 7. Axios
- 역할: 프론트엔드에서 백엔드 API로 HTTP 요청을 보내는 라이브러리
- 왜 필요한가:
- React에서 Express 서버로 Todo 목록 조회, 추가, 수정, 삭제 요청을 보내기 위해 필요하다.
- `GET`, `POST`, `PUT`, `DELETE` 요청을 비교적 읽기 쉽게 작성할 수 있다.
- `fetch`로도 가능하지만 초보자에게는 Axios가 응답 처리 측면에서 더 편한 경우가 많다.

### 8. dotenv
- 역할: `.env` 파일의 환경변수를 불러오는 라이브러리
- 왜 필요한가:
- MongoDB 연결 문자열 같은 민감한 정보를 코드에 직접 쓰지 않기 위해 필요하다.
- 보안과 관리 측면에서 중요하다.
- 개발 환경과 배포 환경의 설정을 분리하기 쉽다.

### 9. cors
- 역할: 다른 주소에서 오는 요청을 허용하도록 설정하는 미들웨어
- 왜 필요한가:
- 로컬 개발에서는 프론트엔드와 백엔드가 서로 다른 주소에서 실행될 가능성이 크다.
- 예를 들어 프론트는 `localhost:5173`, 백엔드는 `localhost:5000`에서 실행될 수 있다.
- 이때 브라우저가 요청을 막지 않도록 CORS 설정이 필요하다.

### 10. nodemon
- 역할: 백엔드 파일이 수정될 때 서버를 자동으로 재시작하는 개발 도구
- 왜 필요한가:
- 코드를 수정할 때마다 서버를 수동으로 껐다 켤 필요가 없어진다.
- 개발 속도가 빨라지고 반복 작업이 줄어든다.

### 11. CSS
- 역할: 화면의 색상, 여백, 크기, 배치 등을 꾸미는 스타일 언어
- 왜 필요한가:
- 기능만 있는 화면이 아니라 사용자가 보기 쉬운 UI를 만들기 위해 필요하다.
- Todo 앱의 입력창, 버튼, 목록 정렬 등을 다듬을 수 있다.

### 12. Tailwind CSS
- 역할: 미리 정해진 클래스 이름으로 빠르게 스타일을 적용하는 CSS 프레임워크
- 왜 필요한가:
- CSS를 처음부터 길게 작성하지 않고도 빠르게 화면을 꾸밀 수 있다.
- 과제처럼 제한된 시간 안에 UI를 정리할 때 유용하다.
- 다만 CSS 기초를 익히고 싶다면 기본 CSS만 사용해도 충분하다.

### 13. Vercel
- 역할: 프론트엔드와 일부 백엔드 기능을 배포할 수 있는 클라우드 플랫폼
- 왜 필요한가:
- 과제 제출 조건에 배포 URL이 포함되어 있다.
- GitHub와 연동하면 배포가 비교적 쉽다.
- 프론트엔드를 실제 웹 주소로 공개할 수 있다.

### 14. Git
- 역할: 코드 변경 내역을 관리하는 버전 관리 도구
- 왜 필요한가:
- 작업 과정을 저장하고 이전 상태를 추적할 수 있다.
- 실수했을 때 변경 이력을 확인하기 좋다.
- GitHub에 코드를 올리기 위해 필요하다.

### 15. GitHub
- 역할: Git 저장소를 온라인으로 관리하는 서비스
- 왜 필요한가:
- 과제 제출에 저장소 URL이 필요하다.
- 코드 백업과 공유가 가능하다.
- Vercel과 연동해 자동 배포하기 좋다.
